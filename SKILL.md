# shuiye-ui

把需求转成可验证的浏览器原型。优先保证核心用户路径、页面状态和交互反馈完整，再处理视觉细节；原型用于沟通和验证，不默认视为生产代码。

## 调用方式

```text
$shuiye-ui
$shuiye-ui edit <页面或文件> <修改需求>
$shuiye-ui env
$shuiye-ui preview [端口]
$shuiye-ui help
$shuiye-ui upgrade <优化想法>
```

## 请求路由

1. 无参数：从任务和需求生成或补全原型。
2. `edit`：修改已有页面、组件、状态或交互。
3. `env`：检测原型运行环境并给出最小安装建议。
4. `preview`：启动本地静态服务器并验证入口。
5. `help`：显示精简帮助。
6. `upgrade`：维护本 skill，不修改业务项目。

## 资源路由

- `references/ui-methods.md`：分析信息架构、响应式设计、交互状态、无障碍和质量验收时读取。
- `references/preview-troubleshooting.md`：执行 `env`、`preview` 或排查资源、路由、端口问题时读取。
- `assets/templates/interaction-flow.md`：需要新建或重构原型交互说明时使用。

不要为了普通页面修改读取全部参考资料，只加载当前任务需要的部分。

## 项目定位

从当前目录向上定位项目根目录，优先识别：

```text
<project-root>/doc/task.md
<project-root>/doc/project.md
<project-root>/prototype/
```

规则：

- 生成原型时优先读取 `doc/task.md`，按需补读 `doc/project.md`。
- 若任务文档不存在但 PRD 足够明确，可从 PRD 分析；两者都不存在时停止并说明缺少输入。
- 需要先生成任务时提示使用 `$shuiye-task`，不要虚构页面需求。
- 修改或预览时可以只依赖现有 `prototype/`。
- 先检查现有项目技术与目录约定，不强制套用固定文件结构。

## 生成原型

### 1. 分析需求

读取任务、PRD 和已有原型，提取：

- 用户角色、目标和核心使用场景。
- P0/P1 页面及其来源任务。
- 页面入口、出口、返回路径和异常路径。
- 各页面的加载、空、成功、错误、禁用等状态。
- 目标设备、视口、输入方式和无障碍要求。
- 已有品牌、组件、字体、颜色和资源约束。

明确区分文档事实、从代码发现的约定和设计假设。缺少关键决策时只询问会显著改变页面结构或交互的问题。

### 2. 设计方案

生成前输出精简方案：

```text
UI 原型方案
===========
输入依据: <任务/PRD/已有原型>
目标设备: <移动端/桌面端/响应式>
核心路径:
- <入口 -> 操作 -> 结果>
页面清单:
- <优先级> <页面>：<目标和关键状态>
目录与技术:
- <沿用项目约定或建议结构>
设计假设:
- <尚未确认的假设>
验证:
- <预览和验收方式>
```

首次创建多个页面、会覆盖现有文件或方案存在关键假设时，等待确认。用户已明确要求直接生成且没有覆盖风险时可继续。

### 3. 实现

- 所有输出写入 `<project-root>/prototype/`。
- 优先沿用已有 HTML、CSS、JavaScript 和组件约定。
- 没有既有结构时使用简洁的静态多页或单页结构，以满足交互为准。
- 使用语义化 HTML、集中式设计变量和可理解的 class 名称。
- 链接、按钮、表单和关键状态必须真实可操作，避免只有静态截图效果。
- 页面之间使用可解析的相对路径；入口页能到达所有核心流程。
- 使用真实或明确标注的示例文案，避免无意义占位文本。
- 优先复用项目已有图片和图标；新增外部素材前确认来源和许可。
- 不添加后端、真实支付、真实认证或第三方生产凭证。需要数据时使用清晰隔离的模拟数据。
- 原型说明确有价值时，使用 `assets/templates/interaction-flow.md` 生成或更新说明；不要机械创建 README。

### 4. 验证

至少执行：

1. 枚举页面和入口，检查文件及资源路径。
2. 启动本地 HTTP 服务器，不以 `file://` 作为唯一验收方式。
3. 打开入口并走通每条 P0 用户路径。
4. 检查导航、返回、按钮、表单和弹层。
5. 检查目标视口及至少一个相邻断点。
6. 检查加载、空、错误和成功反馈中适用的状态。
7. 检查控制台错误、404 和明显布局溢出。
8. 汇报生成文件、访问地址、覆盖范围和未验证项。

可用浏览器自动化时，优先进行截图和关键路径点击验证。不要声称未实际执行的浏览器测试已经通过。

## 编辑原型

用于 `$shuiye-ui edit <页面或文件> <修改需求>`。

1. 根据页面名、标题、路由和文件名定位目标；有多个匹配时列出候选。
2. 读取目标文件及其直接依赖的共享样式、脚本和导航。
3. 说明当前行为、目标行为、影响文件和回归风险。
4. 覆盖现有设计或影响多个页面时先展示修改方案并确认。
5. 实施最小完整修改，保持已有视觉语言和代码结构。
6. 同步受影响的导航、共享组件和交互说明。
7. 重新验证目标场景及相邻页面，不只检查文件是否写入。

不得用整页重写掩盖一个局部修改，也不得删除用户已有交互或内容，除非需求明确要求。

## 环境检测

用于 `$shuiye-ui env`。

- 检查 `prototype/`、入口文件、现有包管理器、Node.js 和可用静态服务器。
- 先利用项目已有命令，再考虑 `npx`、Python 静态服务器或 `live-server`。
- 只推荐完成当前任务必需的工具。
- 安装软件、全局 npm 包或浏览器组件前说明用途、命令和影响并获得确认。
- Figma、云服务、令牌和付费服务不是默认依赖。

具体选择和排障读取 `references/preview-troubleshooting.md`。

## 本地预览

用于 `$shuiye-ui preview [端口]`。

1. 确认 `prototype/` 和入口存在。
2. 检查目标端口；未指定时选择可用的常见开发端口。
3. 优先使用项目已有预览命令，否则选择环境中已存在的静态服务器。
4. 启动后用 HTTP 请求验证入口返回成功。
5. 需要打开 GUI 浏览器时申请相应权限。
6. 报告访问地址、服务命令、进程状态和停止方式。

不要在命令退出后把服务报告为“运行中”，也不要占用已有服务端口。

## 质量门槛

- P0 页面和用户路径与任务来源可追溯。
- 页面间无死链，主要操作有结果反馈。
- 关键状态完整，不只展示理想成功态。
- 布局适配目标设备，无明显横向溢出或遮挡。
- 键盘焦点、表单标签、对比度和触摸目标达到原型可用水平。
- HTML/CSS/JavaScript 易于阅读，没有不必要的框架和依赖。
- 原型中的模拟行为、限制和未实现能力明确可见。

详细标准读取 `references/ui-methods.md`。

## 安全边界

- 覆盖或批量删除 `prototype/` 文件前必须展示清单并确认。
- 不修改 `doc/project.md` 或 `doc/task.md`；发现需求差异时报告并建议由对应 skill 维护。
- 不访问真实用户数据，不写入 API 密钥、访问令牌或个人凭据。
- 不自动安装全局工具，不默认调用外部设计或素材服务。
- 不把模拟登录、支付、上传或删除描述为真实业务能力。
- 预览服务器只绑定当前任务需要的接口；暴露到局域网或公网前确认。

## Help

```text
shuiye-ui
=========
生成原型: $shuiye-ui
修改页面: $shuiye-ui edit <页面或文件> <修改需求>
检测环境: $shuiye-ui env
本地预览: $shuiye-ui preview [端口]
帮助:     $shuiye-ui help
升级:     $shuiye-ui upgrade <优化想法>

默认输入: doc/task.md，按需读取 doc/project.md
默认输出: prototype/
```

## Upgrade

用于 `$shuiye-ui upgrade <优化想法>`。

1. 定位实际安装的项目级或用户级 skill。
2. 读取 `SKILL.md` 和本次改动涉及的资源。
3. 区分明确低风险修改与探索性想法。
4. 外部服务、费用、隐私、安装、部署或破坏性改动先说明风险、最小试验和成功标准。
5. 需要确认时，确认后先做最小试验或静态检查。
6. 修改前备份受影响文件。
7. 保持中文入口简洁，把详细规则路由到标准资源。
8. 保留生成、编辑、环境和预览业务行为。
9. 验证 frontmatter、资源路由、安全规则和至少一个代表场景。

## 验收清单

- frontmatter 只有 `name` 和 `description`。
- 默认、`edit`、`env`、`preview`、`help` 和 `upgrade` 可发现。
- 输入缺失、覆盖文件、安装工具和启动服务有明确处理。
- 入口指向所有可选资源。
- 生成场景可追溯到任务或 PRD。
- 编辑场景保留现有结构并执行回归检查。
- 预览场景实际验证 HTTP 入口。
- 官方 skill 校验通过。
