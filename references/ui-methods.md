## 分析框架

从四个角度建立设计依据：

- 用户：用户是谁、要完成什么、在什么环境下操作、主要阻碍是什么。
- 产品：核心价值、优先级、信息层级、业务规则和成功标准是什么。
- 视觉：现有品牌、层级、颜色、字体、间距和资源约束是什么。
- 技术：目标设备、浏览器、已有框架、性能和可维护性限制是什么。

先引用任务或 PRD 中的事实，再记录设计假设。不要把常见设计习惯当成已确认需求。

## 信息架构与用户路径

每条核心路径至少包含：

1. 入口：用户从哪里开始。
2. 决策：用户需要理解或选择什么。
3. 操作：用户执行的主要动作。
4. 反馈：系统如何表明处理中、成功或失败。
5. 结果：用户如何确认目标完成。
6. 恢复：失败、取消、返回和重试如何处理。

页面清单按 P0、P1、P2 排序。P0 应覆盖最小闭环，不要因追求页面数量而拆分无价值页面。

## 页面和组件设计

- 用稳定布局和重复模式帮助用户建立预期。
- 主操作在视觉与交互层级上清晰，次要操作不过度竞争。
- 表单标签长期可见，不仅依赖 placeholder。
- 危险操作与普通操作区分，并提供确认或撤销。
- 组件优先复用，但不要为了抽象而增加原型复杂度。
- 导航应支持当前位置识别、返回和核心入口到达。

可借鉴原子设计的层级：基础元素、组合控件、业务区块、页面模板和具体页面。只有真实复用时才抽取组件。

## 响应式设计

根据目标设备决定断点，不机械使用固定数值。没有项目约定时可从以下范围试验：

- 小屏移动端：320px 至 428px。
- 平板或窄桌面：768px 左右。
- 桌面端：1024px 至 1440px。

检查：

- 内容不产生意外横向滚动。
- 固定头部和底部不遮挡内容。
- 触摸目标建议不小于约 44px。
- 移动端不依赖 hover。
- 桌面端可点击元素有 hover 和 focus 反馈。
- 文本放大后仍可读，布局不会立即破坏。

移动优先适合移动产品，但不是所有项目的默认答案。后台系统或大屏工具应从主要使用场景出发。

## 交互状态与反馈

根据页面用途选择适用状态：

- 初始：用户尚未操作。
- 加载：说明正在处理，避免重复提交。
- 空状态：解释为何为空并提供下一步。
- 成功：明确结果和后续动作。
- 错误：说明原因、影响和恢复方式。
- 禁用：说明当前不可用原因。
- 部分数据：允许用户理解哪些内容有效。
- 离线或超时：提供重试或保存策略。

反馈应及时、明确且与动作位置接近。动画用于解释状态变化，不应阻碍操作；优先使用 `transform` 和 `opacity`，并尊重减少动态效果偏好。

## 无障碍

原型至少做到：

- 使用语义化标题、导航、主内容、按钮和表单元素。
- 输入控件具有关联标签和可理解的错误说明。
- 所有核心操作可通过键盘完成。
- 焦点样式清晰，弹层打开和关闭后焦点位置合理。
- 不只依靠颜色传递成功、错误或选中状态。
- 文本与背景具有可辨识对比度。
- 图片根据作用提供替代文本；装饰图片可为空替代文本。
- 点击和触摸区域足够大。

## 视觉与内容

- 建立少量设计变量：颜色、字体、间距、圆角、阴影和层级。
- 间距使用一致尺度，例如 4 或 8 的倍数，但服从已有设计系统。
- 正文保持易读行高和行宽，标题层级不跳跃。
- 文案使用用户语言描述动作，例如“保存更改”，避免含糊的“确定”。
- 示例数据应贴近业务，并标注模拟或演示性质。
- 颜色含义保持一致，避免同一颜色同时表示不同状态。

## 技术实现

- HTML 结构语义化，避免大量无意义容器。
- CSS 优先复用变量和现有类，不堆叠内联样式。
- JavaScript 只实现验证交互需要的逻辑。
- 事件处理避免污染全局，重复初始化应保持幂等。
- 资源路径使用可从预览根目录解析的方式。
- 不为静态原型引入大型框架，除非项目已使用或交互确实需要。
- 图片应控制尺寸；动画避免频繁触发布局计算。
- 模拟 API 和本地存储应能重置，避免污染其他场景。

## 验收清单

功能：

- 所有 P0 页面可从入口到达。
- 页面间跳转、返回和取消可用。
- 按钮、表单、弹层和错误恢复可操作。
- 无明显死链、404 或控制台异常。

视觉：

- 对齐、间距、字体和颜色保持一致。
- 主次层级清楚，内容不被固定区域遮挡。
- 目标视口和相邻断点无明显溢出。

可用性：

- 用户无需猜测下一步操作。
- 操作后有加载、成功或错误反馈。
- 键盘导航、焦点和标签达到原型验证要求。

技术：

- 文件结构和命名符合项目约定。
- 共享样式与脚本没有无意破坏其他页面。
- HTTP 预览和关键路径已实际检查。
